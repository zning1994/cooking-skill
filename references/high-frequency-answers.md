# High-Frequency Answers

This file defines stable answer patterns for the most common user questions.

Use this file before broad recipe generation when the user asks a direct technique, rescue, or substitution question.

## 1. Technique cluster

### Beef tenderness

Question patterns:

- 牛肉怎么炒嫩一点
- 牛肉一炒就老怎么办

Answer pattern:

1. cut against the grain
2. light marinade with soy + starch + oil
3. high heat, short cook
4. remove when just color-changed

### Crispy potato shreds

Question patterns:

- 土豆丝怎么炒脆
- 土豆丝总是发黏怎么办

Answer pattern:

1. rinse starch after slicing
2. soak briefly, drain fully
3. hot pan, quick stir
4. season near the end

### Non-dry chicken thigh

Question patterns:

- 鸡腿怎么做不柴
- 鸡腿肉口感发柴怎么避免

Answer pattern:

1. sear first for color
2. then braise / stew
3. avoid long hard boil
4. use doneness cues, not fixed long overcook

## 2. Rescue cluster

### Too salty

Question patterns:

- 做咸了怎么救

Answer pattern:

1. add neutral bulk (potato/tofu/vegetables)
2. rebalance with small water if dish allows
3. avoid blind sugar compensation

### Too sour tomato dish

Question patterns:

- 番茄太酸怎么调回来

Answer pattern:

1. extend cooking for integration
2. add small sugar incrementally
3. add egg / starch / stock to soften sharpness

### Too watery

Question patterns:

- 炒出来总是出水
- 汤汁太稀怎么办

Answer pattern:

1. raise heat and reduce
2. split batches next time
3. use light starch slurry only when dish type fits

## 3. Substitution cluster

### No cooking wine

Question patterns:

- 没有料酒怎么办

Answer pattern:

1. use ginger + scallion + blanching logic
2. state that aroma is not fully equivalent
3. adjust seasoning order to avoid raw smell

### No starch

Question patterns:

- 没有淀粉怎么做

Answer pattern:

1. reduce liquid
2. increase reduction time
3. accept thinner finish when needed

### No wok

Question patterns:

- 没有炒锅怎么办
- 平底锅能做吗

Answer pattern:

1. smaller batches
2. hotter pan surface
3. reduce expectation for wok-hei

## 4. Answer quality checklist

For high-frequency answers, ensure:

1. direct answer first
2. no unnecessary clarification if intent is clear
3. one failure point called out explicitly
4. one fallback provided when relevant
5. avoid performance-heavy filler
